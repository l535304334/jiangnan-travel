## 一、项目概况

**江南出行**是一个面向江西省的网约车智慧服务平台，包含 **乘客端、司机端、管理后台** 三个前端 + 一个 Spring Boot 后端，深度集成 **5 个 AI 引擎**（客服、预测、调价、调度、风控）和 **高德地图**。

项目在 **Trae IDE + AI 编程助手** 的协助下，用 **4 周（20 个工作日）** 从零完成。

## 四、4 周开发历程

### Week 1 — 基础搭建 + 用户模块 + AI客服
- Maven项目骨架 + MyBatis-Plus + Spring Security + JWT
- 21 张表 DDL + 种子数据
- 用户注册/登录（密码+验证码双模式）
- AI 智能客服（DeepSeek，注入江西文旅知识库，含降级方案）
- AI 出行预测引擎（按时段统计 + 新用户冷启动）
- 收藏地址 CRUD
- **交付**：75 个源文件，8 个 API，编译 6.7s，启动 4.9s

### Week 2 — 订单核心 + AI调度 + 动态调价
- 阶梯计价引擎（0-3km / 3-50km / 50-200km / 200+km 四档）
- 动态调价引擎（供需比算法，上限 1.5x）
- AI 智能调度引擎（距离 0.5 + 评分 0.3 + 空闲 0.2 多因素加权）
- Redisson 分布式锁（防止重复接单）
- 订单全生命周期（创建→接单→到达→开始→完成→支付→评价）
- 优惠券系统（领取/使用/新用户发券）
- **交付**：99 个源文件，21 个 API

### Week 3 — 前端三端 + WebSocket + 高德地图 + 实时追踪
- Vue 3 项目搭建（Vite 5.4，5.42s 构建）
- 三合一登录页（乘客/司机/管理员切换，密码+验证码双模式）
- 24 个页面组件，27 条路由
- WebSocket 实时推送（司机位置 `/ws/driver-location` + 订单状态 `/ws/order/{id}`）
- 高德地图组件 AmapView（20 个江西地标坐标库 + Haversine 距离计算）
- AI 客服悬浮窗（DeepSeek 对话）
- 取消订单 R2 风控 + 一键报警接口
- **交付**：前端 36 个文件，1708 个 Vite 转换模块

### Week 4 — 集成测试 + 优化 + 文档 + Git 部署
- Axios 拦截器修复（业务错误统一 reject + ElMessage 提示）
- IP 限流（60 次/分钟/ip，Redis 滑动窗口）
- Redis 缓存（4 级 TTL：车型 24h / 地标 12h / 寄语 12h / 热点 30m）
- 11 条 SQL 复合索引（含地理查询、高频查询）
- 5 个集成测试（UserServiceTest）
- 管理后台 10 个后端接口
- AI 数据分析助手、城市文化短句、司机收入统计、疲劳提醒 R7
- 20 地标模糊搜索 + 下单页自动坐标匹配
- 行程追踪页文化短句 5 秒轮播
- Git 本地仓库 + post-commit 自动 CHANGELOG + GitHub 推送
- **最终交付**：109 个后端源文件 + 36 个前端源文件

## 五、关键技术决策

| 决策 | 原因 |
|------|------|
| 包名 `com.jiangnan.travel`（非`smarttravel`） | 开发中简化，避免过长包名 |
| 密码登录为主要认证方式 | 验证码依赖运营商，沙箱环境不可用 |
| 高德地图配置 `_AMapSecurityConfig` | 高德 JS API 2.0 要求安全密钥 |
| 地标坐标本地化（20 个预设点） | 避免高德地理编码 API 额外调用 |
| 风控仅实现 R2+R4+R7 | 按 MVP 可降级计划，其余为生产级扩展 |
| Post-commit Hook 自动 CHANGELOG | bash 脚本，每次 `git commit` 后触发 |

## 十二、从用户与产品经理视角的可行性判断

### 12.1 用户价值

对乘客来说，项目的核心价值是"江西本地化出行 + 文旅信息 + 基础网约车服务"。如果仅作为毕业实习项目或企业内部原型，已经可以清楚展示登录、下单、估价、追踪、优惠券、客服和后台管理的完整闭环。

对司机来说，价值在于简化接单、订单状态流转和收入查看。不过司机端当前仍偏演示，真正落地前还需要补齐司机认证材料、车辆资料、在线时长、位置上传可靠性、订单拒单/取消规则等细节。

对运营人员来说，管理后台已经能看用户、司机、订单和风控告警，但还缺少强运营能力，例如筛选、导出、详情审核、车型价格调整、优惠券发放、异常订单处理等。

### 12.2 商业与落地可行性

| 维度 | 判断 | 原因 |
|------|------|------|
| 学术/实习交付 | 高 | 技术栈完整、功能闭环清晰、文档充分，适合展示软件工程实践过程 |
| 企业原型验证 | 中高 | 能支撑业务流程演示，但真实运营前需要合规、支付、地图、短信、运维、安全全面补齐 |
| 真实商业上线 | 中低 | 网约车业务强监管，必须具备运营许可证、司机/车辆资质、资金结算、隐私合规、风控审计等能力 |
| 本地化差异化 | 中高 | 江西文旅融入是亮点，但还需要更具体的城市/景区/线路运营场景来拉开差异 |

### 12.3 产品风险

1. 合规风险：真实网约车业务不能只靠技术系统上线，必须先满足平台、车辆、司机资质要求。
2. 供需冷启动：没有司机供给时，乘客下单体验会断裂；需要模拟派单或运营侧种子司机方案。
3. 地图依赖：高德 Key、服务开通、域名白名单、配额都会影响演示稳定性。
4. 支付与发票：当前为模拟支付，电子发票只是表结构/占位，不能支撑真实交易。
5. AI 宣称边界：项目写了 5 个 AI 引擎，但部分能力实际是规则算法或静态数据，应在文档中明确 MVP 口径，避免过度承诺。
